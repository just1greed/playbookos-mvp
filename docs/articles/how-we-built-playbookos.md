# 24 小时、2235 次请求、432M Token：我们如何用 Codex 搭起 PlaybookOS

日期：2026-03-09  
工具：`Codex`  
模型：`GPT-5.4`  
参考统计：总耗时约 `24h`，总请求数约 `2235` 次，总 Token 约 `432M`

## 引子

如果只看今天的仓库状态，PlaybookOS 已经像一个相对完整的系统了：

- 有控制面 API
- 有本地持久化
- 有 SOP ingestion
- 有 Task / Run / Reflection 主链
- 有全局看板和工作台式前端
- 还能让 OpenClaw 这类外部 agent 通过 `manifest / context / intake / apply` 接入

但它并不是先有一套完美设计，再一次性落地出来的。

它是沿着一条很明确的工程主线，一步一步搭起来的：先把最有价值的链路跑通，再把系统从 demo 推向控制面，再让它逐渐具备被人类和 agent 共同操作的能力。

## 第一步：先确认我们做的不是聊天 UI，而是工作控制面

这次搭建过程中，最早确认的一件事其实不是技术选型，而是产品定位。

PlaybookOS 不应该只是一个对话界面，也不应该只是“帮你调用几个工具”的 Agent 壳子。它要解决的是一个更底层的问题：

> 如何把目标、SOP、能力、执行、验收、复盘和知识沉淀统一到同一个系统里，并且让这些状态都可以被追踪、审批、恢复和持续改进。

所以从一开始，我们就把系统核心主链定义为：

`Goal -> SOP -> Skill / MCP -> Task -> Run -> Acceptance / Reflection / Knowledge`

这条链非常重要，因为后面无论是 API、对象模型、前端结构还是 agent 接入，基本都是围绕它来长出来的。

## 第二步：先立控制面骨架，而不是急着做“聪明功能”

一旦目标明确，接下来最关键的就不是“让 AI 看起来更智能”，而是先把系统最小骨架搭起来。

我们优先确定了一批必须存在的控制面对象：

- `Goal`
- `Playbook`
- `Skill`
- `MCPServer`
- `Task`
- `Run`
- `Session`
- `Acceptance`
- `Reflection`
- `KnowledgeBase`
- `KnowledgeUpdate`
- `Event`
- `StoredObject`

然后让这些对象都能落在一个实际可运行的本地系统上：

- API：FastAPI
- 存储：SQLite
- 对象文件：本地 object store

这一步听起来不惊艳，但它非常关键。因为一旦对象模型和状态流先稳定下来，后面的 AI 能力、前端体验、自动化治理才有真正的附着点。

## 第三步：决定从 SOP 进入，而不是从自由聊天进入

如果一个系统想真正帮助用户搭建工作流，最有效的输入往往不是一句模糊需求，而是一份真实 SOP。

所以在往前推进时，我们很快收敛出一个重要策略：

> 先做 `Markdown-first`，把最稳的一条 SOP 导入链跑通，而不是一开始就承诺 PDF、Docx、图片、多附件全支持。

这让产品边界瞬间变得清晰很多。

我们要先解决的，不是“支持多少种格式”，而是：

- 能不能把一份 Markdown SOP 解析成结构化步骤
- 能不能识别这份 SOP 需要哪些工具域
- 能不能判断系统里已有的 Skill / MCP 是否够用
- 能不能告诉用户还缺哪些 Skill / MCP
- 能不能引导用户去补齐这些能力

于是，PlaybookOS 的 ingestion 主链逐步成型：

`Markdown SOP -> 提取步骤 -> 识别工具域 -> 生成 Skill 建议 -> 识别 MCP 缺口 -> 产出提示词 -> 物化 draft Skill / draft MCP`

从这一刻开始，系统就不再只是“保存 SOP”，而是开始真正理解 SOP 背后需要的能力结构。

## 第四步：从文档系统走向执行系统

仅仅理解 SOP 还不够。一个工作操作系统如果不能把 SOP 往下推动到执行层，价值仍然非常有限。

所以我们继续往下搭：

- `Playbook -> Task DAG` 规划
- 任务依赖推进
- Run 创建
- Supervisor / Worker Session
- `waiting_human` 与审批状态联动
- Reflection 与 KnowledgeUpdate 的学习闭环

这一步非常像从“文档系统”跨向“运行系统”的分水岭。

它意味着：

- Playbook 不只是文件，而是可编排的执行模板
- Task 不只是状态，而是依赖图中的工作单元
- Run 不只是日志，而是执行尝试与审批节点
- Reflection 不只是备注，而是会回流进知识与未来能力设计的输入

也就是从这里开始，PlaybookOS 才真正具备了“OS”的味道。

## 第五步：把前端从临时页面，重构成工作台系统

随着对象越来越多、主链越来越长，原来的单页式前端很快就不够用了。

这时候我们又做了一个关键决策：

> 用户打开系统后，首先看到的应该是一个全局看板，而不是某个局部编辑器。

于是前端结构开始重构为：

- 默认首页：全局看板
- 左侧导航：各类工作台
- 设置页：模型与全局治理入口

之后，导航被逐步拆成：

- 目标
- SOP
- Skill
- MCP
- 知识库
- 任务
- 会话
- 自动迭代
- 审批流
- 提示词
- 模型设置
- 全局设置
- 会话管理

首页也逐步被补成多个关键区块：

- 系统摘要
- 流程总览
- 关键阻塞
- 推荐动作
- 进行中任务
- 自动迭代动态

这轮改造最大的意义，是让系统的信息架构终于和它的控制面定位匹配了起来。

## 第六步：把“设置”做成真实能力，而不是装饰面板

很多内部系统都会卡在这一步：页面上有“设置”入口，但实际上只是摆设。

我们这轮搭建时，特意把它做成了真实能力：

### 模型设置

模型配置不再完全依赖环境变量，而是具备：

- 运行时设置存储
- provider preset
- 连接测试
- 命名环境 profile
- 最近成功 probe 记录

### 全局设置

全局设置开始控制：

- 默认语言
- 默认路由
- 默认筛选
- 自动刷新
- 环境标签

### 会话管理

会话不再只是展示摘要，而可以：

- 按 Goal / Run 过滤
- 修改状态
- 编辑标题与总结

这一轮之后，PlaybookOS 的控制台不再只是“可以看”，而是“可以管”。

## 第七步：确认系统必须给 Agent 用，而不只是给人用

在项目推进中，有一次非常关键的方向转向：

> PlaybookOS 大概率不会只给人类看，它还会被 OpenClaw 这类外部 agent 当作控制面来使用。

这意味着，仅有 Dashboard 远远不够。

系统必须提供一套 agent-friendly 的操作面。

于是我们落地了：

- `GET /api/agent/manifest`
- `GET /api/agent/context`
- `POST /api/agent/intake`
- `POST /api/agent/apply`
- `GET/POST/PUT /api/delegation-profiles*`
- `skills/playbookos-operator/`

这个设计非常关键，因为它把 agent 的操作方式也标准化了：

1. 先读 `manifest`，理解系统对象和推荐工作流
2. 再读 `context`，同步当前系统状态
3. 再用 `intake` 把自然语言请求或 Markdown SOP 转成 dry-run 操作计划
4. 最后在 delegation 约束下用 `apply` 执行一部分 builder 动作

这样，外部 agent 不需要“猜 API”，而是可以被有边界地接入系统。

## 第八步：持续回头审文档，让文档服从真实实现

在快速迭代里，一个非常常见的问题是：文档开始跑在代码前面。

我们这轮过程中其实多次停下来做了这件事：

- 审 README
- 审功能边界
- 审文案是否夸大
- 审交互是不是暗示了还没做完的能力

最典型的一次，就是把对外能力明确收敛到 `Markdown-first`，避免让人误判“多格式 SOP 已经正式支持”。

后来我们又进一步把根说明文档拆成：

- `README.md`：完整中文
- `README-EN.md`：完整英文

并新增了：

- `scripts/openclaw_demo.sh`

让 OpenClaw / 外部 agent 的接入不只停留在文字说明，而是能直接跑一个完整演示。

## 第九步：用工程事实，而不是模糊回忆，记录这条建设路径

当一个系统在很短时间内持续推进时，最容易丢失的不是代码，而是“为什么这么做”的上下文。

所以我们又补上了：

- `docs/records/progress.md`
- `docs/records/iteration-memory.md`
- `docs/records/building-playbookos-story.md`

这些记录的价值，不只是留档，而是让后续每一次迭代都能知道：

- 哪些决策已经确认
- 哪些能力是当前主线
- 哪些缺口是故意延后，而不是忘了做

这也是为什么今天我们还能把整个过程比较清楚地还原出来。

## 这轮搭建最值得保留的四条方法论

回过头看，这次最重要的并不只是“做了哪些功能”，而是逐渐稳定下来的四条方法论：

### 1. Runnable-first

先让主链跑通，再补治理细节。

### 2. Markdown-first

先把最稳的一条 SOP 导入链做深，而不是过早扩格式。

### 3. Agent-first interface

外部 agent 不应直接猜系统内部接口，而应通过：

`manifest -> context -> intake -> apply`

### 4. Documentation follows implementation

文档必须服从真实实现，不能把未完成的未来能力写成既成事实。

## 从现在往后，最重要的改进方向是什么

虽然 PlaybookOS 已经有了一个可运行的 MVP，但它接下来最重要的工作并不是“继续堆功能”，而是把几条已经形成的主线做深做稳：

- 把 Markdown SOP 的工具识别、Skill 引导、MCP 缺口提示继续做稳
- 把 agent 托管链从 builder 第一版推进到更可治理的执行面
- 把 MCP 从“登记 + probe”推进到“runtime + credential + tool execution”
- 把错误日志、自迭代、issue 快捷提交串成闭环
- 把前端工作台继续补齐到更适合长期运营的控制台形态

也就是说，这个系统当前最需要的不是“看起来更大”，而是“主链更稳、治理更强、协作更自然”。

## 结尾

如果用一句话总结这次 24 小时的搭建过程，那就是：

> 我们不是先画出一个完美蓝图，再按图施工；而是围绕一条最有价值的主链，持续迭代，把一个想法一步步推成了一个可运行、可演示、可继续生长的系统。

而在这个过程中，Codex 与 GPT-5.4 更像是一个高频协作的工程搭档：

- 快速探索方案
- 实时改代码与补文档
- 在产品边界和工程现实之间反复对齐
- 让“想法”尽快变成“可以跑的东西”

今天的 PlaybookOS 还远不是终局，但它已经拥有了一个很重要的品质：

它不再只是一个想法，也不只是一个 demo，而是一条真实存在、已经跑起来的工程主线。
